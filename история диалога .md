## Общая картина

Проект представляет собой аналитическую платформу для анализа экономических показателей России на основе данных Росстата. На данный момент реализован **один индикатор — ИПЦ** (Индекс потребительских цен), два других (безработица, ключевая ставка ЦБ) числятся как заглушки "скоро". Проект состоит из **двух независимых частей**: Python-аналитика и React-фронтенд.

---

## 1. Python-аналитика (бэкенд/скрипты)

### `sarima_forecast.py` — автономный скрипт прогнозирования

Запускается из командной строки (`python sarima_forecast.py`). Делает следующее:

1. Загружает исторические данные ИПЦ из `output/ipc_monthly.csv` (419 наблюдений, с января 1991 по ноябрь 2025)
2. Обучает модель **SARIMA(1,0,1)(1,0,1,12)** — модель временных рядов с сезонностью 12 месяцев
3. Генерирует прогноз на 12 месяцев вперёд с 95% доверительными интервалами
4. Выводит таблицу прогноза в консоль
5. Сохраняет результат в `output/forecast_sarima.csv`
6. Строит график matplotlib и сохраняет в `output/sarima_forecast.png`

### `app.py` — интерактивный Streamlit-дашборд

Полноценное веб-приложение на Streamlit с Plotly-графиками. Запускается через `streamlit run app.py`. Возможности:

- **Сайдбар**: настройка параметров SARIMA (p, d, q, P, D, Q, s), горизонта прогноза (3-36 мес.), фильтрация по датам
- **Метрики**: текущий ИПЦ, среднее за год, прогноз на след. месяц, AIC модели, объём данных
- **4 вкладки**:
  - **Прогноз** — график с фактическими данными + прогноз + 95% доверительный интервал + таблица
  - **Исторические данные** — линейный график + тепловая карта ИПЦ по годам/месяцам
  - **Диагностика** — ACF, PACF, Q-Q plot, гистограмма остатков, остатки во времени
  - **Данные** — таблица с фильтрацией по годам и поиском + скачивание CSV

Стилизован кастомным CSS с градиентными карточками метрик, тёмной темой сайдбара.

## 2. React-фронтенд (основное веб-приложение)

### Технологический стек

- **React 19** + **Vite 7** (сборка)
- **React Router v7** (роутинг)
- **Recharts** (интерактивные графики)
- **CSS Variables** (тёмная тема)

### Роутинг (`App.jsx`)

| Путь | Компонент | Назначение |
|------|-----------|------------|
| `/` | `Home` | Главная страница — каталог индикаторов |
| `/indicator/cpi` | `IndicatorPage` | Детальная страница ИПЦ |
| `/indicator/:code` | `StubIndicator` | Заглушка для остальных индикаторов |

### Главная страница (`Home.jsx`)

- Hero-секция с заголовком и бейджами
- Сетка из 3 карточек-индикаторов (данные из `indicators.json`): ИПЦ (активен), безработица и ключевая ставка (неактивны, помечены "Скоро")
- Секция "О проекте" с технологическим стеком

### Страница ИПЦ (`IndicatorPage.jsx`)

- **Метрики**: текущее значение, предыдущий месяц, прогноз на декабрь 2025
- **Интерактивный график** (`InteractiveChart.jsx`) — Recharts с переключением периодов (1 год / 5 лет / 10 лет / все данные), фактические данные + прогнозная линия + доверительный интервал, кастомный тултип
- **Информационные карточки**: описание показателя, методология, статистика (максимум/минимум/среднее)
- **Таблица прогноза** на 12 месяцев с доверительными интервалами
- **Таблица исторических данных** (`DataTable` компонент)

### Данные — статические JSON-файлы

Фронтенд **не обращается к бэкенду**. Все данные вшиты в статические JSON-файлы:

- `src/data/cpi.json` — полный набор: метаданные, текущее значение, прогноз, 468 исторических точек (1991–2025)
- `src/data/indicators.json` — каталог из 3 индикаторов с текущими значениями

## Находки

### 1. Скрипты в `росстат/`

Главный файл — `rosstat_collector.py`. Он содержит точные URL-ы для автоматической загрузки данных с Росстата:

```python
SOURCES = {
    "price_indexes_monthly": {
        "url": "https://rosstat.gov.ru/storage/mediabank/ipc_mes_11-2025.xlsx",
    },
    "price_indexes_weekly": {
        "url": "https://rosstat.gov.ru/storage/mediabank/Nedel_ipc.xlsx",
    },
    "price_weekly_report": {
        "url": "https://rosstat.gov.ru/storage/mediabank/195_17-12-2025.html",
    }
}
```

А `rosstat_downloader.py` имеет ещё более полный список из 8 файлов с разной детализацией (помесячные, квартальные, недельные и т.д.).

### 2. Критическая проблема: URL меняется каждый месяц

URL содержит суффикс с месяцем и годом: `ipc_mes_**11-2025**.xlsx` -> `ipc_mes_**12-2025**.xlsx`. Это значит, что жёстко прописать URL нельзя — нужен алгоритм:
- Генерировать URL динамически по текущей дате
- Пробовать несколько вариантов (текущий месяц, предыдущий)
- Fallback: парсить страницу-индекс `rosstat.gov.ru/free_doc/new_site/prices/ipc.htm` для получения актуальной ссылки

### 3. SSL-сертификаты

Все существующие скрипты используют `verify=False` — это работает, но не enterprise-level. У тебя есть 4 сертификата:

- `russiantrustedca.pem` — RSA цепочка 2022 (root + sub, действует до 2032)
- `russiantrustedca2024.pem` — RSA цепочка 2024 (root + sub, действует до 2029)
- `russian_trusted_root_ca_gost_2025.cer` — ГОСТ корневой (DER, до 2040)
- `russian_trusted_sub_ca_gost_2025.cer` — ГОСТ подчинённый (DER, до 2030)

Для Python requests нужны PEM-файлы. **`russiantrustedca2024.pem`** — основной рабочий сертификат. ГОСТ-сертификаты Python не поддерживает нативно (нужна библиотека `gost-crypto`), но для HTTP-запросов к Росстату RSA-сертификаты работают.

### 4. CertIZDAT репозиторий

Этот проект — **исследовательский** инструмент для анализа использования российских CA-сертификатов на сайтах рунета. Он **не помогает** нам непосредственно. Однако он подтверждает подход: нужно добавить российские CA-сертификаты в trusted bundle для HTTP-клиента. Наше решение: создать объединённый CA-бандл (системные CA + российские CA) и использовать его в requests.

### 5. Для серверного деплоя (TimeWeb)

Нужно:
- Включить PEM-файлы в Docker-образ бэкенда
- Создать combined CA bundle: `cat /etc/ssl/certs/ca-certificates.crt russiantrustedca2024.pem > combined_ca_bundle.pem`
- Настроить env-переменную `REQUESTS_CA_BUNDLE=/app/certs/combined_ca_bundle.pem`

## Часть 1: Как работает автообновление данных

Это реальный и рабочий механизм. Объясняю по шагам.

### Что происходит внутри

Когда бэкенд стартует, FastAPI запускает **APScheduler** — это встроенный в Python планировщик задач (аналог cron, но внутри приложения):

```39:42:/Users/iprofi/tradingeconomics/rosstat/backend/app/main.py
        scheduler.add_job(
            daily_update_job,
            trigger=CronTrigger(
                hour=settings.scheduler_cron_hour,
```

Каждый день в **6:00 по Москве** шедулер вызывает функцию `daily_update_job()`. Вот что она делает:

**Шаг 1 — Поиск файла на Росстате.** `RosstatFetcher` перебирает URL-ы вида `https://rosstat.gov.ru/storage/mediabank/ipc_mes_MM-YYYY.xlsx`, начиная с текущего месяца и идя назад до 6 месяцев. Для каждого URL делает HTTP HEAD запрос (лёгкий, без скачивания файла). Если Росстат отвечает 200 — файл найден.

**Шаг 2 — Скачивание.** Скачивает найденный Excel-файл. Проверяет, что это реально XLSX (magic bytes `PK`), а не HTML-страница с ошибкой (Росстат возвращает 200 даже когда файла нет).

**Шаг 3 — Парсинг.** Открывает нужный лист Excel (01 для общего ИПЦ, 02 для продуктов и т.д.), парсит pivot-таблицу, извлекает все помесячные значения ИПЦ.

**Шаг 4 — Upsert в БД.** Каждая точка данных вставляется с `ON CONFLICT DO NOTHING` — если такая дата уже есть, дубликат игнорируется. Подсчитывается, сколько **новых** записей добавилось.

**Шаг 5 — Переобучение модели.** Если есть хотя бы 1 новая запись → запускается `train_and_forecast()` (OLS Multi-Window), генерируется новый прогноз на 12 месяцев, старый помечается как `is_current=False`.

**Шаг 6 — Сброс кэша.** Redis-кэш для этого индикатора инвалидируется → следующий запрос пользователя получит свежие данные.

### Это реально работает?

Да, абсолютно. Это стандартный паттерн в продакшене:
- **APScheduler** — зрелая библиотека (100M+ скачиваний), используется в продакшене повсеместно
- Шедулер живёт **внутри** процесса uvicorn, не нужен отдельный cron или celery
- Если сервер перезагрузится — Docker перезапустит контейнер (`restart: unless-stopped`), uvicorn стартует, шедулер зарегистрирует задачу заново
- Если Росстат недоступен — ошибка логируется, старые данные и прогноз остаются, следующий день попробует снова

## Cloud Server (VPS) vs App Platform на Timeweb

### App Platform — почему НЕ подходит для нашего проекта

App Platform (PaaS) — это решение для stateless-приложений. Ты пушишь код, платформа сама собирает и деплоит. Звучит удобно, но вот что сломается:

1. **PostgreSQL и Redis** — App Platform не даёт тебе контейнеры с базами. Придётся покупать отдельно **Managed PostgreSQL** (~500-800₽/мес) и **Managed Redis** (~300-500₽/мес) как отдельные сервисы. Это дороже и сложнее в настройке.

2. **APScheduler (ежедневный ETL)** — App Platform может масштабировать инстансы, перезапускать их, спавнить новые. APScheduler живёт в памяти процесса. Если платформа запустит 2 инстанса бэкенда — ETL запустится **дважды** одновременно. Если перезапустит контейнер между 5:59 и 6:01 — задача пропущена.

3. **Docker Compose** — App Platform не поддерживает `docker-compose.yml`. Придётся настраивать каждый сервис отдельно (backend, frontend) и связывать через их UI.

4. **Сертификат Росстата** — у нас он встроен в Docker-образ через `COPY`. В App Platform нужно разбираться, как пробросить файл (через build args, или volume, или environment — зависит от платформы).

5. **Entrypoint-логика** (миграции + seed) — может работать криво, т.к. PaaS часто перехватывает `ENTRYPOINT`.

### Cloud Server (VPS) — почему подходит

Наш `docker-compose.yml` — это **полностью самодостаточная единица деплоя**. Один файл описывает 4 сервиса, их связи, healthcheck-и, volumes, restart policies. На VPS это просто работает.

| Критерий | VPS | App Platform |
|---|---|---|
| Цена | ~500-700₽/мес за всё | ~1500-2000₽ (app + managed DB + managed Redis) |
| Сложность деплоя | 1 команда: `docker compose up -d --build` | Настройка 3-4 отдельных сервисов через UI |
| APScheduler | Работает из коробки | Проблемы с дублированием/пропуском |
| docker-compose.yml | Нативная поддержка | Не поддерживается |
| SSL-сертификат Росстата | Встроен в образ | Нужны workaround-ы |
| Контроль | Полный (логи, SSH, мониторинг) | Ограниченный |
| Масштабирование | Ручное (поднять тариф) | Автоматическое |

### Вердикт

**VPS — однозначно правильный выбор** для нашего проекта. App Platform решает другую задачу: stateless API или SPA без фонового процессинга. У нас stateful-система с базой, кэшем, шедулером и специфичным SSL. Весь стек упакован в `docker-compose.yml` и заточен под VPS.

App Platform имел бы смысл, если бы у нас был простой REST API без фоновых задач, без Redis, с внешней базой. Но это не наш случай.

**Cursor**

Потому что `ipc_monthly.csv` (seed-данные) был собран в ноябре 2025 — это начальная загрузка. Новые данные подтянутся автоматически завтра в 6:00 утра по Москве, когда шедулер впервые сработает.

Но можно запустить обновление прямо сейчас. Введи на сервере:

```bash
docker compose exec backend python -c "
import asyncio
from app.tasks.scheduler import daily_update_job
asyncio.run(daily_update_job())
"
```

Это вручную запустит тот же ETL, который будет работать каждый день автоматически — скачает свежий файл с Росстата, добавит декабрь 2025 и январь 2026 (если доступен), переобучит модель.

import asyncio, logging

from app.database import async_session

from sqlalchemy import select
async def retrain():
    async with async_session() as db:
        for code in ['cpi','cpi-food','cpi-nonfood','cpi-services']:
            ind = await db.execute(select(Indicator).where(Indicator.code==code))
            i = ind.scalar_one_or_none()
            if not i: continue
            data_q = await db.execute(select(IndicatorData).where(IndicatorData.indicator_id==i.id).order_by(IndicatorData.date))
            all_data = data_q.scalars().all()
            if len(all_data) < 36: continue
            dates = [d.date for d in all_data]
            values = [float(d.value) for d in all_data]
            result = train_and_forecast(dates, values, forecast_steps=12)
            old_q = await db.execute(select(Forecast).where(Forecast.indicator_id==i.id, Forecast.is_current.is_(True)))

            fc = Forecast(indicator_id=i.id, model_name=result.model_name, model_params={'cumulative_12m': result.cumulative_12m}, aic=result.aic, bic=result.bic, is_current=True)

            await db.flush()
            for fp in result.points:

## Шаг 1 — Настроить DNS на nic.ru

1. Зайди в личный кабинет nic.ru
2. Найди свой домен → **Управление зоной DNS** (или "DNS-серверы / Управление записями")
3. Добавь **две A-записи**:

| Тип | Имя | Значение |
|---|---|---|
| A | @ | 5.129.204.194 |
| A | www | 5.129.204.194 |

`@` — это сам домен (например `example.ru`), `www` — это `www.example.ru`.

4. Сохрани. DNS обновится за 5-30 минут (иногда до 2 часов).

## Шаг 2 — Проверить DNS

Через 10-15 минут проверь с мака:

```bash
dig +short ТВОЙ-ДОМЕН.ru
```

Должно показать `5.129.204.194`. Если да — переходи к шагу 3.

## Шаг 3 — Установить HTTPS на сервере

Подключись к серверу и поставь Caddy (автоматический HTTPS):

```bash
ssh root@5.129.204.194
```

```bash
apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | tee /etc/apt/sources.list.d/caddy-stable.list
apt update && apt install -y caddy
```

Потом настрой Caddy (замени `ТВОЙ-ДОМЕН.ru` на свой реальный домен):

```bash
cat > /etc/caddy/Caddyfile << 'EOF'
ТВОЙ-ДОМЕН.ru, www.ТВОЙ-ДОМЕН.ru {
    reverse_proxy localhost:80
}
EOF
```

```bash
systemctl restart caddy
```

Caddy **автоматически** получит SSL-сертификат от Let's Encrypt. Через 10-20 секунд сайт будет доступен по `https://ТВОЙ-ДОМЕН.ru`.
