# Widget-Handoff — бриф для ЛК (вход менеджера в чат на сайте)

> **Создан:** 2026-06-10. Дополняет `MANAGER_HANDOFF_CONTRACT.md` (модель v2.1/v2.2).
> **Назначение:** перенести handoff (перехват диалога менеджером) с канала MAX на
> канал **widget** (виджет на сайте). Логика та же; отличается только **доставка**
> сообщений клиенту: в MAX бот пушит сам, в виджете **сервер→браузер пуша нет**,
> поэтому виджет **поллит** историю.
> **Раздел работ:** MGP-backend — сделано (этот документ — контракт). Фронтенд
> виджета и операторский UI для виджет-диалогов — на стороне ЛК.

---

## 0. Что уже готово на стороне MGP (задеплоено, под флагом)

Бэкенд-часть реализована и **инертна**, пока в конфиге `OPERATOR_HANDOFF_CHANNELS`
не добавлен `widget`. Включение — поэтапно (см. §4). Канал-независимы и уже
работают: триггеры (`book_click`/`phrase`/`contact`/`booking_intent`), события
`manager_alert`/`operator_inbound`, «тихий» handoff (клиенту на триггере ничего не
шлём), пауза ИИ только при реальном заходе менеджера, «поздняя отмена», авто-возврат.

**Гейт виджета (раздельно с MAX):**
- `OPERATOR_HANDOFF_ENABLED=true` (общий рубильник).
- `OPERATOR_HANDOFF_CHANNELS` должен включать `widget`.
- `OPERATOR_HANDOFF_WIDGET_ALL_TENANTS=true` → фича для **всех** виджет-ассистентов;
  иначе — точечный `OPERATOR_HANDOFF_WIDGET_ASSISTANT_IDS` (CSV) для обкатки.
- MAX по-прежнему гейтится своим `OPERATOR_HANDOFF_ASSISTANT_IDS` — виджет его не задевает.

## 1. Доставка в виджет = polling `/api/runtime/history`

### 1.1 Запрос (из браузера-виджета, та же авторизация, что и `/api/v1/chat`)
```
GET /api/runtime/history?conversation_id=<session_id>&after_id=<int|optional>
```
- Без `after_id` — последние сообщения (полное восстановление, как раньше).
- С `after_id` — только сообщения с `id > after_id` (дёшево для частого поллинга).

### 1.2 Ответ (расширен для handoff)
```json
{
  "conversation_id": "…",
  "messages": [
    { "id": 12345, "role": "user|assistant|operator",
      "content": "…", "tour_cards": [], "created_at": "ISO" }
  ],
  "operator_mode": false,
  "handoff_state": "none|requested|operator|returned",
  "operator_actor": null
}
```
- **`role:"operator"`** — реплика менеджера. Рисовать как сообщение человека
  («Менеджер»), отдельным стилем от ИИ.
- **`operator_mode:true`** — менеджер за рулём, ИИ на паузе → показать баннер
  «С вами менеджер» и **участить поллинг** (например 2–3с вместо 5–8с).
- **`id`** — монотонный (autoincrement). Виджет хранит max(id) и шлёт как `after_id`.

### 1.3 Рекомендация по поллингу
- Диалог открыт: поллить `?after_id=<lastId>` каждые ~5–8с.
- `operator_mode=true` (или `handoff_state in [requested, operator]`): ~2–3с.
- Новые `messages` дорисовывать; обновлять `lastId`.

## 2. Ответ менеджера и перехват (LK → MGP, тот же back-channel, что и MAX)

Заголовок: `X-MGP-Service-Token: <MGP_OPERATOR_TOKEN>` (тот же, что уже выдан).
База — `https://max.navilet.ru` (домен общий для runtime; маршрутизация по
`conversation_id`, не по каналу).

- `POST /api/runtime/operator/message` — `{ conversation_id, text, operator:{id,name} }`
  - **widget:** MGP **не** шлёт во внешний канал, а записывает `role='operator'`
    в БД → виджет заберёт поллингом. Ответ `200 {status:"sent", message_id}`.
- `POST /api/runtime/operator/handoff` — `{ conversation_id, action:"take|return", operator }`
  - `take` → `operator_mode=true` (ИИ на паузе). Для виджета «менеджер на связи»
    показывает **баннер** (из `operator_mode`), отдельного сообщения нет.
  - `return` → `operator_mode=false` (ИИ снова ведёт диалог).

Ошибки: `401` неверный токен · `403` handoff выключен для ассистента/канала ·
`404` диалог не найден · `409` неизвестный канал.

## 3. Сообщения клиента при активном менеджере + события в ЛК

- Когда `operator_mode=true` и клиент пишет в виджет (`/api/v1/chat`), бэкенд
  **подавляет ИИ** и возвращает `{ "reply":"", "operator_mode":true, "suppressed":true }`.
  Фронт виджета: показать сообщение клиента как отправленное и НЕ рисовать пустой
  пузырь ИИ (можно «менеджер ответит»). Само сообщение уедет менеджеру событием
  `operator_inbound` (как в MAX).
- `manager_alert` / `operator_inbound` приходят в ЛК тем же приёмником, что и для
  MAX. У виджета `external_*` пустые → имя в уведомлении будет «Клиент» (ожидаемо).

## 4. Порядок включения (безопасно)

1. **MGP backend задеплоен** — виджет инертен (`CHANNELS=max`). ✅
2. **ЛК делает фронтенд виджета** (поллинг §1.3, рендер `operator`, баннер,
   обработка `suppressed`) и операторский UI для виджет-диалогов в дашборде
   (переиспользовать из MAX, для `channel='widget'`).
3. Когда ЛК готов → MGP включает виджет **точечно на МГП основном офисе**
   (`assistant_id=593471b7-…`, компания «МГП Тур»/`mgp-tour`) → e2e-тест.
4. Валидировали → MGP ставит `WIDGET_ALL_TENANTS=true` → раскатка на всех виджетах.
