# Тестовые сценарии для проверки 15 исправлений AI-ассистента

## Как тестировать

Запустите интерактивный тестер в терминале:

```bash
cd "/Users/lukiansilagadze/Desktop/AI-ASSIANT PROD"
./test_chat.sh
```

Для каждого сценария ниже:
1. Наберите `new` чтобы начать свежий диалог
2. Вводите сообщения по порядку (столбец **Вы**)
3. Проверяйте ответ бота (столбец **Ожидаемое поведение**)
4. Набирайте `logs` для просмотра серверных логов

Логи также доступны:
- Серверные: `logs/server_*.log`
- Диалоговые: `logs/dialogue_*.md`
- Docker: `docker logs mgp-local-test-backend-1 --tail 100`

## СЦЕНАРИЙ 4 — Каскад при 0 результатов (Fix #3, #5, #9)

**Цель:** При 0 результатов бот автоматически расширяет фильтры по каскаду, каждый раз делая НОВЫЙ `search_tours`, а не переопрашивая старый `requestid`.

| # | Вы | Ожидаемое поведение |
|---|---|---|
| 1 | `Тур на Кубу из Москвы, 3-10 мая, 2 взрослых, 5 звезд, ультра все включено` | Бот ищет. Если 0 результатов: |
| — | (ожидаем автоматический каскад) | **Шаг 1:** Расширяет даты +7 дней, делает НОВЫЙ `search_tours` |
| — | | **Шаг 2:** Если опять 0 → убирает meal, НОВЫЙ `search_tours` |
| — | | **Шаг 3:** Если опять 0 → убирает stars, НОВЫЙ `search_tours` |
| — | | **КРИТИЧЕСКАЯ ПРОВЕРКА:** Бот должен СООБЩИТЬ что именно изменил: «Расширил даты поиска...», «Убрал ограничение по питанию...» |

**Что проверяем в логах:**
- Каждый шаг каскада — отдельный вызов `search_tours` (не `get_search_status` по старому `requestid`)
- Нет строки `POLL LIMIT` (если есть — safety net сработал правильно, не давая переопрашивать)
- Бот НЕ выдаёт «Произошла временная ошибка»

## СЦЕНАРИЙ 14 — Двойное сообщение (Fix #12)

**Цель:** На одно сообщение пользователя бот должен отвечать ровно одним сообщением.

Этот сценарий сложнее проверить из командной строки — он был связан с race condition при параллельных запросах. Для проверки:

```bash
# Отправьте 2 запроса одновременно с одним conversation_id:
CONV="ваш_conversation_id"
curl -s -X POST http://localhost:8088/api/v1/chat \
  -H "Content-Type: application/json" \
  -d "{\"message\": \"Привет\", \"conversation_id\": \"$CONV\"}" &
curl -s -X POST http://localhost:8088/api/v1/chat \
  -H "Content-Type: application/json" \
  -d "{\"message\": \"Привет\", \"conversation_id\": \"$CONV\"}" &
wait
```

**Ожидание:** Один запрос обработается, второй подождёт (mutex). В логах: `SESSION LOCK` для одного из запросов.
